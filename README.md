# Пример внедрения микрофронтов на Module Federation (Vite) с описанием

Реализация микрофронтов с Module Federation (Vite) в монорепозитории на примере дашборда виджетов (список товаров, курс валют, прогноз погоды, график продаж).

**Полезные инструменты для монорепозитория:**

- Turborepo — умное кэширование и параллельный запуск
- pnpm workspaces — эффективное управление зависимостями
- Lerna — классический инструмент для монорепозиториев

# Корень приложения

Корень приложения содержит package.json, в котором сборка хоста и ремоутов осуществляется с помощью Turborepo (либо Lerna, Nx). В корневом package.json указаны скрипты, позволяющие запустить отдельно или скоупом все части приложения. Для работы с Turbo желателен pnpm или yarn. Для pnpm обязателен pnpm-workspace.yaml, из которого будут считываться нужные директории.
Версия Node - 24.11.1 (с иными более старыми версиями могут быть конфликты).
В turbo.json настройки работы сборщика, указания папок.

# Хост

Настроена конфигурация хоста в vite.config.ts:

- в shared указываем папки и библиотеки, которыми делимся, задаем версии зависимостей
- в remotes указываем список ремоутов с портами и енвиками
- указываем порты, на которых запускать хост-приложение (server, preview).

В package.json и vite.config.ts указываем имя пакета - на него завязан turbo-скрипт в корне приложения.
В package.json указываем ключевые зависимости директории, версию и shared пакеты зависимостей, где "workspace:\*" является протоколом, который используется в монорепозиториях, и означает "возьми пакет @mfe-dashboard/shared-ui не из интернета (npm registry), а из моего локального монорепозитория, и используй ту версию, которая там есть". Совместим с pnpm. Аналог workspace - "file:../../packages/shared-ui", где указываем что нужно взять файл из определенного пути.

Хост содержит в src свою бизнес-логику.
Добавляем в гитигнор билды в dist и assets.

# Ремоуты

Настроена конфигурация хоста в vite.config.ts:

- exposes, который определяет, какие именно модули (компоненты, функции, файлы) ваше remote-приложение делает доступными для других приложений (хоста). В нем задаем доступ.
- в shared задаем версии зависимостей, в т.ч. к shared
- порты ремоута в server, preview.

В package.json и vite.config.ts указываем имя пакета - на него завязан turbo-скрипт в корне приложения.
В package.json указываем ключевые зависимости директории, версию и shared пакеты зависимостей, где "workspace:\*" является протоколом, который используется в монорепозиториях, и означает "возьми пакет @mfe-dashboard/shared-ui не из интернета (npm registry), а из моего локального монорепозитория, и используй ту версию, которая там есть". Совместим с pnpm. Аналог workspace - "file:../../packages/shared-ui", где указываем что нужно взять файл из определенного пути.

Ремоут содержит в src свою бизнес-логику.
Добавляем в гитигнор билды в dist.

Каждый ремоут определен в remotes хоста в vite.config.ts
Прописываем декларационные модули ремоутов в хосте (types/remote-modules.d.ts, vite-env.d.ts).
В корневом package.json корректируем скрипты сборки и разработки с указанием нужных ремоутов.

# Shared utils, UI, API, stores

Настроена конфигурация каждого shared (vite.config.ts):

- Экспорты - что экспортируется в хост/ремоут
- Проставляется версия.
- В name записано имя текущего пакета, который добавлен в package ремоута и хоста.
- В package.json каждого shared присутствует экспорт "./package.json": "./package.json", иначе чревато ошибками чтения remoteEntry.js

Shared содержит в src свою бизнес-логику.

# Скрипты

- **"dev":** "turbo run dev",
- **"dev:all":** "concurrently \"pnpm dev:host\" \"pnpm dev:remote\"",
- **"dev:host":** "turbo run dev --filter=@mfe-dashboard/host --parallel",
- **"dev:api":** "json-server --watch db.json --port 3001",
- **"dev:remote":** "turbo run preview --filter=@mfe-dashboard/remote-products --parallel",
- **"build:remote":** "turbo run build --filter=@mfe-dashboard/remote-products",
- **"build":** "turbo run build",
- **"preview":** "turbo run preview",
- **"type-check":** "turbo run type-check",
- **"lint":** "turbo run lint",
- **"clean":** "rm -rf node*modules apps/*/node*modules packages/*/node*modules apps/*/dist packages/\_/dist"

**pnpm install** - установка зависимостей в каждой директории

**Remove-Item -Recurse -Force apps/host/node_modules/.vite -ErrorAction SilentlyContinue** - для очистки кэша в хосте, затем pnpm i в корне.

Прежде чем запустить сервера, нужно сбилдить ремоут: pnpm build:remote,
затем запустить pnpm dev:all. При каждом изменении в коде ремоута, нужно билдить его и перезагружать страницу.

Для работы запускаем через pnpm dev:all и pnpm dev:api (запуск json-server).

# Варианты организации микрофронтендов

## 1. **Микрофронты в монорепозитории**

Один репозиторий, содержащий все микрофронтенды (хост и все ремоты) в отдельных папках внутри apps.

### Структура:

```
vue-mfe-monorepo/
├── packages/
│ ├── host/ # Главное приложение (порт 5000)
│ ├── remote1/ # Микрофронтенд 1 (порт 5001)
│ ├── remote2/ # Микрофронтенд 2 (порт 5002)
│ └── remote3/ # Микрофронтенд 3 (порт 5003)
├── package.json # Общие зависимости и скрипты (workspaces)
├── lerna.json # (Опционально) для Lerna
└── turbo.json # (Опционально) для Turborepo
```

Плюсы:

✅ Единое управление версиями

✅ Простой локальный запуск всех приложений одновременно

✅ Легко делить общие типы и утилиты

✅ Упрощенный код-ревью (все изменения в одном PR)

✅ Интеграция с инструментами монорепозиториев (Turborepo, Lerna, pnpm workspaces)

Минусы:

❌ Меньшая автономность команд (все в одном репозитории)

❌ При разрастании может замедляться

## 2. **Мультирепозиторий**

Каждое приложение (хост и каждый ремот) — в отдельном репозитории.

## Структура:

```
github.com/company/
├── host-app-repo/ # Хост приложение
├── remote-catalog-repo/ # Микрофронтенд каталога
├── remote-cart-repo/ # Микрофронтенд корзины
└── remote-auth-repo/ # Микрофронтенд авторизации
```

Плюсы:

✅ Максимальная автономность команд

✅ Независимое версионирование и деплой

✅ Каждая команда выбирает свой стек и процессы

Минусы:

❌ Сложнее настраивать локальную разработку

❌ Требуется координация версий между репозиториями

❌ Дублирование конфигураций
